# Archives（归档）默认文件夹模板

> 适用于：已完成的项目、停止维护的领域
> 使用说明：归档时保持原有结构，便于后续查找

---

## 📁 默认文件夹结构

```
Archives（归档）/
├── Projects/             ← 已完成的项目
│   └── {年份}/
│       └── {项目名}_归档于{YYYYMMDD}/
└── Areas/                ← 停止维护的领域
    └── {年份}/
        └── {领域名}_归档于{YYYYMMDD}/
```

---

## 📋 归档规则

### 项目归档

**归档时机：**
- 项目目标已达成
- 项目已上线/交付
- 项目终止或取消

**归档命名：**
```
Projects/{年份}/{项目名}_归档于{YYYYMMDD}/
```

**示例：**
```
Archives/Projects/2026/AI清予网站_归档于20260710/
```

### 领域归档

**归档时机：**
- 领域停止维护
- 领域转型或更名
- 领域合并到其他领域

**归档命名：**
```
Areas/{年份}/{领域名}_归档于{YYYYMMDD}/
```

**示例：**
```
Archives/Areas/2026/朋友圈经营_归档于20260710/
```

---

## ⚠️ 归档注意事项

### 1. 保持原有结构
归档时，整个文件夹原封不动移动到归档区，不要删除或修改内部文件。

**正确做法：**
```
移动前：Projects/AI清予网站/
移动后：Archives/Projects/2026/AI清予网站_归档于20260710/
```

### 2. 添加归档后缀
统一使用 `_归档于{YYYYMMDD}` 后缀，便于识别归档时间。

### 3. 按年份分类
使用年份文件夹，避免归档区过于混乱。

### 4. 写归档说明
在归档文件夹内添加 `归档说明.md`，记录：
- 归档原因
- 项目/领域状态
- 重要成果
- 经验教训

**模板：**
```markdown
# 归档说明

**归档日期：** 2026-07-10
**归档原因：** 项目已上线，进入维护阶段

## 项目状态
- ✅ 已完成核心功能开发
- ✅ 已通过测试验收
- ✅ 已正式上线

## 重要成果
- 上线日期：2026-07-01
- 用户数：500+
- 核心功能：...

## 经验教训
- ...
```

## 📌 归档后的使用

### 查找归档内容
1. 按年份浏览：`Archives/Projects/2026/`
2. 按名称搜索：使用 Glob 或 Grep 工具

### 恢复归档项目
如果需要重新启动项目：
1. 从归档区复制到 Projects/
2. 去掉 `_归档于{YYYYMMDD}` 后缀
3. 更新项目概览和时间线

### 彻底删除
- 归档超过 2 年且确认不再需要的项目
- 删除前再次确认
- 建议先压缩备份到云盘

---

## 🎯 归档的意义

1. **保持工作区整洁** - 只显示正在进行的项目和领域
2. **便于复盘总结** - 归档时沉淀经验教训
3. **历史可追溯** - 完整保留项目历史
4. **减少决策负担** - 不用每次都看到已完成的项目
