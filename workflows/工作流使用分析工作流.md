# 工作流使用分析工作流

> 识别哪些工作流被频繁使用且稳定，建议封装成Skill
> 版本：v1.0

---

## 触发时机

- 用户说"分析工作流"
- 用户说"哪些工作流可以变成Skill"
- 定期执行（建议每月1次）

---

## 完整执行流程

### 步骤1：询问分析范围

```
要分析哪个时间范围的工作记录？
[1] 过去7天
[2] 过去30天
[3] 过去3个月
[4] 自定义时间范围

请选择：
```

**时间范围计算**：
- 过去7天：今天往前推7天
- 过去30天：今天往前推30天
- 过去3个月：今天往前推90天
- 自定义：用户指定开始和结束日期

### 步骤2：扫描工作记录

读取指定时间范围内的所有工作记录：
```
Areas/{领域}/04-工作记录/{领域}工作记录_{年月}.md
Projects/{项目}/04-工作记录/{项目}工作记录_{年月}.md
Resources/04-工作记录/Resources工作记录_{年月}.md
```

**扫描范围**：
- 如果时间范围跨多个月，读取所有相关月份的工作记录文件
- 例如：2026-06-15 ~ 2026-07-06，需要读取 202606 和 202607 两个月的记录

### 步骤3：提取工作流调用信息

从工作记录中识别以下模式：

**识别关键词**：
- "执行了XXX工作流"
- "按照XXX工作流完成"
- "调用XXX工作流"
- "读取XXX工作流"
- "使用XXX工作流"

**提取信息**：
- 工作流名称
- 调用日期
- 调用位置（哪个领域/项目）

**统计维度**：
- 调用次数
- 首次调用日期
- 最后调用日期
- 调用频率（次/天 = 调用次数 / 时间范围天数）
- 调用位置分布

### 步骤4：分析稳定性

对每个工作流判断稳定性：

**检查版本历史**：
- 读取 `05-版本历史/01-工作流/` 文件夹
- 查看工作流在分析时间范围内是否有版本更新
- 格式：`v{版本号}_{日期}_{文件名}.md`

**稳定性判断**：
- ✅ **稳定**：期间无版本更新，或只有1次更新且在时间范围开始前
- ⚠️ **不稳定**：期间有2次或以上版本更新
- ❌ **频繁变动**：期间有5次或以上版本更新

### 步骤5：判断是否适合封装

**封装标准**：
1. **高频使用**：调用次数 ≥ 5次
2. **稳定**：期间无版本更新或只有少量更新
3. **通用性**：在多个位置被调用（可选）

**分类**：
- ✅ **建议封装**：高频 + 稳定
- 📦 **已封装**：已经是Skill的工作流
- ⏳ **观察中**：高频但不稳定
- 📉 **低频使用**：调用次数 < 5次

### 步骤6：生成分析报告

```markdown
📊 工作流使用频率分析报告

**分析时间范围**：{开始日期} ~ {结束日期}（{X}天）
**扫描位置**：{Y}个领域 + {Z}个项目
**发现工作流**：{N}个

---

## ✅ 建议封装成Skill

### 1. {工作流名称}
- **位置**：{工作流路径}
- **调用次数**：{X}次
- **调用频率**：{Y}次/天
- **首次调用**：{日期}
- **最后调用**：{日期}
- **稳定性**：✅ 期间无版本变更
- **调用位置**：{领域1}、{领域2}
- **建议Skill名称**：/qy-{建议的名称}

---

## 📦 已封装成Skill

- **调用次数**：{X}次

### 2. 结构变更同步工作流

---

### 1. {工作流名称}
- **位置**：{路径}
- **调用次数**：{X}次
- **稳定性**：❌ 期间更新了{Y}次

- **建议**：等稳定后再考虑封装

---

- {工作流3}：1次

---

- 已封装成Skill：{X}个
- 建议新增Skill：{Y}个
- 观察中（需稳定）：{Z}个
- 低频使用：{N}个
```

### 步骤7：询问用户

如果有建议新增的工作流：

```
检测到 {X} 个工作流适合封装成Skill：

1. {工作流名称}（{调用次数}次，稳定）
   建议Skill名称：/qy-{名称}

要现在创建Skill吗？
[1] 是，立即创建
[2] 稍后手动创建
[3] 记录到待办
```

### 步骤8：保存报告（可选）

如果分析范围≥30天，自动保存报告到：
```
Resources/02-系统设计文档/工作流使用分析报告_{年月}.md
```

### 步骤9：记录到用户习惯

在 `Resources/02-系统设计文档/用户习惯记录.md` 中记录本次调用：

```markdown
### {日期} 工作流管理场景

- **用户原话**："{用户说的话}"
- **识别场景**：工作流管理
- **执行的Skill**：/qy-analyze-workflow
- **分析范围**：{时间范围}
- **是否符合预期**：待用户反馈
```

## 特殊情况处理

### 如果没有足够的工作记录

```
⚠️ 工作记录不足

分析时间范围内只有 {X} 天的工作记录。
建议：至少需要7天的工作记录才能进行有效分析。
```

### 如果所有工作流都已封装

```
✅ 所有高频工作流已封装成Skill

当前系统运行良好，无需新增Skill。
```

### 如果没有发现任何工作流调用

```
⚠️ 未发现工作流调用记录

可能原因：
1. 工作记录中未明确记录工作流调用
2. 分析时间范围内没有使用工作流
3. 工作记录格式不符合识别规则

建议：在工作记录中明确记录"执行了XXX工作流"。
```
