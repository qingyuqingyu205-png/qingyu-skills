# 一人公司OS — Claude 使用规则

> **版本：** v3.1  
> **更新日期：** 2026-07-11  
> **重要：** 每次开始工作前必须先读完本文档

---

## 📌 核心原则

1. **先捕获，后分类** - 不确定的内容先放 inbox/，等明确后再分类
2. **优先用 Skill** - 有 Skill 就用 Skill，减少手动操作
3. **永远先确认** - 文件夹操作必须先问用户

---

## 🚫 反模式（Anti-Patterns）

| ❌ 禁止 | ✅ 正确做法 | 原因 |
|---------|------------|------|
| 擅自新建文件夹 | 先列出计划，问用户确认 | 文件夹是结构性决策，不能擅自改 |
| 不确定存储位置时随意选择 | 先放 inbox/，问用户分类 | 错误的位置会导致后续混乱 |
| 直接写入 Areas/Projects | 先写入 inbox/，等用户确认分类 | 减少分类错误 |
| 加载所有文件到上下文 | 只读索引，按需加载 | 上下文窗口有限 |
| 跳过 inbox 直接写 wiki | inbox 是缓冲区，等沉淀后再升级 | 避免半成品文档 |
| 已存在同名文档时不询问就新建 | 询问：更新 or 新建版本 | 避免重复文件 |

## 💬 文件操作规则

**qy-chat-general Skill 专用规则（不适用其他Skill）：**

对于明确指定位置的请求（如"在Projects下创建测试文件夹"）：
- 直接调用工具执行，不需要确认

对于未指定位置的请求（如"创建一个文件夹"）：
- 先询问用户存储位置，再执行

**示例：**
- 用户："帮我在Projects下创建测试文件夹" → 直接调用 create_folder("Projects/测试文件夹")
- 用户："帮我创建一个文件夹" → 询问位置后再执行

| 修改场景 | 修改位置 | 示例 |
|---------|---------|------|
| 系统架构变更（新增/删除核心文件夹） | 根目录 CLAUDE.md | 新增 inbox/ |

| 项目特定规则 | 项目级 CLAUDE.md | 这个项目的命名规范 |

**判断标准：**
- 影响**所有用户/所有项目** → 根目录 CLAUDE.md
- 只影响**某个项目/某个领域** → 子文件夹 CLAUDE.md

**修改流程：**
1. 确定修改范围（根目录 or 子文件夹）
2. 备份旧版本
3. 执行修改
4. 记录到变更日志
5. 调用 `/qy-sync-structure` 检查影响

### 权限实现逻辑

**前端层面：**
```javascript
// 受保护的路径列表
const PROTECTED_PATHS = [
  '/CLAUDE.md',
  '/SYSTEM_DEFAULTS.json',
  '/README.md',
  '/Resources/01-系统工作流',
  '/Resources/02-系统设计文档',
  '/Resources/04-模板库'
];

// 受保护的文件模式
const PROTECTED_FILES = [
  /^\/Resources\/01-系统工作流\/.*\.md$/,
  /^\/Resources\/04-模板库\/.*\.md$/
];

function canUserModify(path) {
  // 检查是否是受保护的路径
  if (PROTECTED_PATHS.some(p => path.startsWith(p))) {
    return false;
  }
  
  // 检查是否匹配受保护的文件模式
  if (PROTECTED_FILES.some(pattern => pattern.test(path))) {
    return false;
  }
  
  return true;
}
```

**用户界面体现：**
- 受保护的文件：右键菜单无"删除"、"重命名"选项
- 受保护的文件：编辑器只读模式，无"保存"按钮
- 受保护的文件夹：无"删除"、"重命名"选项
- 受保护的文件：图标显示🔒标记

## 📚 知识入库规则（v2.0 新增）

### 核心原则

**建立链接 → 产生洞察 → 驱动行动**

1. **建立链接** - 内容与现有项目/知识的连接
2. **产生洞察** - 触发思考、发现关联
3. **驱动行动** - 知识转化为可执行的下一步

### 知识处理流程

**渐进式知识处理（Progressive Summarization）：**
```
第一层：原文保存 → 07-原文存档/
       完整内容 + 源链接，status: processing

第二层：关键高亮 → 在原文中标记重要段落
       status: processing

第三层：个人转写 → 创建知识卡片到 06-知识库/
       用自己的话重述，status: synthesized

第四层：应用链接 → 连接到项目/选题
       建立双向链接，status: applied
```

**关键原则：** 不急于转化，允许多次精炼

### 知识库结构

```
Resources/
├── 06-知识库/              ← 知识管理中心
│   ├── 01-方法论/          ← 核心方法论卡片
│   ├── 02-概念解析/        ← 原子知识卡片
│   ├── 03-工具框架/        ← 可实践的工具/模板
│   ├── 04-行业知识/        ← 行业 specific 知识
│   ├── 05-灵感库/          ← 情绪价值类内容
│   └── 99-待分类/          ← 收集中的笔记
│
├── 07-原文存档/           ← 完整保留输入内容
│   └── [来源]_YYYYMMDD_标题.md
│
└── 08-选题库/             ← 内容创作选题管理
    ├── 待选/
    ├── 进行中/
    └── 已发布/
```

### 使用 /qy-process-input

当需要整理输入内容时，使用 `/qy-process-input` Skill：
1. AI 会读取你的工作记录和日志，了解近期工作
2. 分析内容与现有知识/项目的连接
3. 给出具体的处理建议（保存原文/提炼方法论/应用到项目等）
4. 自动建立双向链接

详细规则见：`Resources/06-知识库/000-知识库规则.md`

### 笔记模板

三个核心模板（位于 `Resources/04-模板库/`）：
- **原文处理模板** - 用于保存和处理输入内容
- **方法论卡片模板** - 用于提炼可复用的方法
- **选题卡片模板** - 用于管理内容创作选题

### 知识回流机制

**当内容写完、用户满意时：**
1. 判断是否有新素材（新案例？新金句？新判断？）
2. 如果有，问用户："这次创作中有 {新素材}，要回填到素材库吗？"
3. 用户同意后，更新到对应位置
